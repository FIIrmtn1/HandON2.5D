# วิธีจัดการทำโครงงาน (HandON2.5D)

คู่มือขั้นตอนการทำงานกับโปรเจกต์เกม 2.5D นี้ใน Unity โดยอ้างอิงจากขั้นตอนจริงที่เคยใช้ทำงาน (เช่น การทำตัวละคร Bendy แบบสไปรต์ 2D บนฉาก 3D)

## 1. ทำความเข้าใจโครงสร้างโปรเจกต์ก่อนแก้ไขอะไร

- `Assets/Scenes/` — ฉากของแต่ละบท เช่น `Ch1_Bedroom.unity`, `Ch3_Kitchen.unity`
- `Assets/Scripts/Player/` — สคริปต์ตัวละคร มีสองชุดขนาน กัน:
  - ชุด 2D: `PlayerController2D`, `HookGrapple2D` ฯลฯ
  - ชุด 3D: `PlayerController3D`, `HookGrapple3D`, `SnapDash3D`, `QuietHang3D` (ไฟล์ที่แก้บ่อยที่สุดของโปรเจกต์)
- `Assets/` (root) — ไฟล์ sprite, animation clip, animator controller ที่ใช้ร่วมกันหลายฉาก เช่น `bendysheet.png`, `bendysheet_0.controller`, `BEndyanim.anim`

**ก่อนแก้ไขอะไร ให้เปิดดูฉากและ hierarchy จริงก่อนเสมอ** อย่าเดาจากชื่อไฟล์อย่างเดียว เพราะแต่ละฉากอาจมี GameObject ที่ไม่เหมือนกัน (เช่น Ch1_Bedroom ไม่มี Table/TV แต่ Ch3_Kitchen มี)

## 2. ระบุปัญหาหรือเป้าหมายให้ชัดก่อนลงมือ

- ถ้าโจทย์กว้างหรือคลุมเครือ ("แก้ภาพเบลอ", "ทำแอนิเมชันให้เดิน") ให้ถามกลับก่อนว่าเป้าหมายจริงคืออะไร แก้ที่ตรงไหน (asset, script, หรือ scene setting)
- แยกให้ออกว่าเป็นปัญหาเรื่อง **เนื้อหา asset** (เช่น ภาพเบลอเพราะไฟล์ภาพเอง) หรือ **การตั้งค่าใน Unity** (เช่น import setting, component) เพราะวิธีแก้ต่างกันคนละทาง

## 3. เตรียม/แก้ไข Asset ก่อนนำเข้าฉาก

- ถ้าต้องแก้ภาพ (เช่น ลบขอบเบลอ, ปรับ alpha): สำรองไฟล์ต้นฉบับก่อนเสมอ (เช่น `bendysheet_original.png`) แล้วค่อยเขียนทับไฟล์จริง เพื่อให้ GUID เดิมยังใช้ได้ sprite slice ที่อ้างอิงอยู่จะไม่พัง
- ถ้าไฟล์ animation clip หรือ animator controller ยังว่างอยู่ ให้สร้างเนื้อหาจริง:
  - Animator Controller: เพิ่ม state, parameter (เช่น bool `IsWalking`), และ transition ระหว่าง state
  - Animation Clip: ใส่ sprite curve ทีละเฟรม กำหนด frame rate ให้เหมาะกับความเร็วที่ต้องการ

## 4. ต่อ Component ให้ตรงกับสิ่งที่ต้องการ

- ตรวจสอบก่อนว่า GameObject ใช้ renderer ชนิดไหนอยู่ (เช่น MeshRenderer ของ Quad กับ SpriteRenderer ใช้งานต่างกัน) ถ้าจะเล่นแอนิเมชันสไปรต์ ต้องมี `SpriteRenderer` + `Animator` ที่ผูกกับ controller ที่ถูกต้อง
- ถ้าต้องย้ายตัวตนตัวละคร (เช่น เปลี่ยนจากโมเดล 3D เป็นสไปรต์ 2D) ให้ย้าย component ที่เกี่ยวกับ gameplay ทั้งหมด (Rigidbody, Collider, PlayerController, tag ฯลฯ) ไปยัง GameObject ใหม่ แล้วค่อยลบของเก่าทิ้ง อย่าลบก่อนย้ายให้ครบ

## 5. แยก Physics Root กับ Visual ออกจากกัน

- อย่าใส่ scale หรือการหมุนที่เป็นเรื่อง "ภาพ" (sprite, billboard) ลงบน root object ที่มี Collider/Rigidbody ตรงๆ เพราะจะกระทบ physics ไปด้วย (ทำให้ตัวละครลอยหรือชนผิดขนาด)
- วิธีที่ถูกต้อง: ให้ root เก็บ scale = (1,1,1) และอยู่ที่ระดับพื้นจริง ส่วน sprite/animator/billboard ให้ย้ายไปอยู่ใน child object (เช่น "Visual") แล้วปรับ scale/ตำแหน่งเฉพาะใน child นั้น

## 6. ผูกพฤติกรรมแอนิเมชันเข้ากับ gameplay จริง

- ถ้าต้องการให้แอนิเมชันเล่นเฉพาะตอนเดิน ไม่ใช่วนตลอดเวลา ให้เขียนสคริปต์เล็กๆ อ่านค่าจาก controller หลัก (เช่น `PlayerController3D.IsQuiet`) แล้วตั้งค่า Animator parameter (`SetBool`) ตามสถานะนั้น
- ถ้า component ที่ต้องอ้างอิงอยู่คนละ GameObject (เช่น อยู่ที่ parent) ให้ใช้ `GetComponentInParent<T>()` แทน `GetComponent<T>()` และเช็ค null ก่อนใช้งานเสมอ

## 7. ทดสอบใน Play Mode ก่อนสรุปว่าเสร็จ

- กด Play แล้วสังเกตจริง: ตำแหน่งตัวละครลอยหรือจมพื้นไหม, แอนิเมชันเล่นถูกจังหวะไหม, ความเร็วแอนิเมชันเหมาะสมไหม
- ถ้าพบปัญหา (เช่น "ลอยจากพื้น" หรือ "แอนิเมชันวนตลอด") ให้กลับไปแก้ที่ต้นเหตุ (มักเป็นเรื่อง scale/hierarchy หรือ state machine) ไม่ใช่แก้ปลายเหตุ

## 8. บันทึกงานอย่างถูกจังหวะ

- **ห้าม save scene ระหว่างอยู่ใน Play Mode** (`manage_scene(action=save)` จะ error ทันที) ต้องออกจาก Play Mode ก่อนเสมอ
- ก่อน save ให้เช็คสถานะ editor (`is_playing:false`, `is_compiling:false`) และตรวจตำแหน่ง/scale ของ object ที่แก้ไขว่านิ่งแล้วจริงๆ
- แก้ไขอะไรระหว่าง Play Mode (เช่น เพิ่ม component, ปรับ transform) จะหายไปทันทีที่ออกจาก Play Mode เพราะ Unity ไม่บันทึกการเปลี่ยนแปลง runtime ให้อัตโนมัติ — ต้องออกจาก Play Mode ก่อนแล้วค่อยแก้ของจริง

## 9. ตรวจสอบผลกระทบก่อนขยายไปฉากอื่น

- ถ้าทำสำเร็จในฉากหนึ่งแล้วจะ "เอาไปใช้กับอีกฉากหนึ่ง" ให้ตรวจ hierarchy ของฉากปลายทางก่อนเสมอ อย่าสมมติว่าโครงสร้างเหมือนกัน (เช่น บางฉากมี object ทดสอบเดิมอยู่แล้วที่ผูก asset ไว้บางส่วน ควรใช้ของเดิมแทนสร้างใหม่)
- ระบุให้ชัดว่าจะ "คัดลอกวิธีการ" หรือ "คัดลอกของเดิมทั้ง object" เพราะสองอย่างนี้ทำให้ผลลัพธ์ต่างกันมาก

## 10. สรุปทุกครั้งก่อนจบงาน

- บอกสั้นๆ ว่าเปลี่ยนอะไรไปบ้าง (ไฟล์ไหน, ฉากไหน, ผลลัพธ์ที่ควรเห็นเมื่อทดสอบ)
- ถ้ามีอะไรที่ยังไม่ได้ทำตามที่ขอ (เช่น ยังไม่ได้ทำอีกฉากหนึ่ง) ให้ระบุไว้ชัดเจน ไม่ปล่อยให้เข้าใจผิดว่าทำครบแล้ว
